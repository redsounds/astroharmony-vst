# AstroHarmony — DSP & System Architecture

## Plugin Type
**VST3 Instrument** (also AU on macOS, Standalone everywhere). No audio input. Generates audio + optional MIDI export.

## Core Components

### Audio thread (C++, real-time)
1. **`Scheduler`** — event-driven beat clock. Two modes:
   - **Host-sync mode:** reads `AudioPlayHead::getCurrentPosition()` every `processBlock`, locks to host BPM + bar position, triggers chord voices on bar boundaries.
   - **Standalone mode:** internal sample-accurate counter using the `tempo` parameter, fires at `(60/bpm) * sampleRate * barsPerChord` intervals.
   - Mode switches automatically based on `isHostPlaying`; user can override via `transport_sync_mode` state.
2. **`juce::Synthesiser`** — 8-voice polyphonic engine. Each voice is a custom `SamplerVoice` wrapping `juce::SamplerSound`.
3. **`SamplerSound` set per instrument** — built once at construction from BinaryData mp3 blobs (decoded via `juce::MP3AudioFormat`). When `active_instrument` changes, the Synthesiser's sound set is swapped.
4. **`juce::dsp::Reverb`** — light cinematic reverb (`roomSize=0.45, damping=0.55, wetLevel=0.14, dryLevel=0.86`), matching the Tone.js `Tone.Reverb({ decay: 1.6, wet: 0.14 })` of the desktop app.
5. **Master gain stage** — multiplies by smoothed `master_volume` (perceptual `(vol/100)^2` taper). 20 ms `SmoothedValue` linear ramp.
6. **Pitch transpose** — affects sampler playback rate by `2^(semitones/12)` on each voice's start note. Smoothed 50 ms to avoid pitch zipper.

### Message thread (C++, non-real-time)
7. **`SessionStore`** — file I/O for `%APPDATA%\AstroHarmony\sessions\*.json`. List/load/save/delete/rename/duplicate. JSON via `juce::JSON` parser.
8. **`MidiExporter`** — receives raw `.mid` bytes from JS, opens `juce::FileChooser`, writes file. The MIDI byte stream is generated by JS-side `lib/midi.ts` (no C++ encoding work).
9. **`EditorBridge`** — thin lifetime-safe bridge between PluginEditor and PluginProcessor for events the editor doesn't need to know about directly (e.g. beat ticks from the audio thread).

### WebView thread (JavaScript, single-threaded UI)
10. **Vite + React 19 single-file bundle**, embedded as one HTML BinaryData resource (proven by Phase 0 spike at 249 KB; expected ~1 MB for full AstroHarmony UI).
11. **Ported modules** from cinematic-composer (read-only references):
    - `lib/theory.ts`, `lib/voicings.ts`, `lib/moodEngine/*`, `lib/midi.ts`, `types/music.ts`
12. **Zustand store** (`lib/store.ts` adapted) — UI-side state. Mirrors the C++ persisted state blob, kept in sync via `pushState` (JS→C++, debounced 200 ms) and `requestInitialState` (C++→JS on WebView load).
13. **React component tree** — all 14 v2 components from `components/v2/*.tsx`.

## Parameter Mapping (C++ DSP wiring)

| Parameter ID | APVTS type | DSP component | C++ wiring |
|---|---|---|---|
| `tempo` | `AudioParameterFloat` | Scheduler (standalone mode only) | `scheduler.setBpm(load())` per block |
| `master_volume` | `AudioParameterFloat` | Master gain stage | `smoothedGain.setTargetValue(perceptual(load()))` |
| `pitch_transpose` | `AudioParameterFloat` | Sampler rate | `smoothedPitch.setTargetValue(load())`; each voice scales playback rate |
| `loop` | `AudioParameterBool` | Scheduler | `scheduler.setLoop(get())` |
| `bypass` | `AudioParameterBool` (host-handled) | n/a | returned via `getBypassParameter()` |

State-blob items (mood, instrument, progression, etc.) bypass APVTS entirely. They mutate via JS native function calls:
- `setInstrument(id)` → swaps `Synthesiser` sound set
- `playProgression(prog, loop)` → loads scheduler with new sequence and starts it
- `pushState(json)` → C++ deserializes, caches the latest authoritative state for `getStateInformation`

## State Persistence Architecture

### `getStateInformation`
1. Snapshot APVTS XML (host-automatable params).
2. Append last-known state blob (the JSON pushed from JS via `pushState`).
3. Combined XML written to `MemoryBlock`.

### `setStateInformation` (DAW project reload)
1. Parse combined XML.
2. Restore APVTS params (host handles automation curves separately).
3. Cache state blob; when the editor opens and WebView calls `requestInitialState`, return this blob to hydrate Zustand.

### `SessionStore` (separate file system, cross-project)
- Independent of DAW state — lives at `%APPDATA%\AstroHarmony\sessions\*.json` (Windows) / `~/Library/Application Support/AstroHarmony/sessions/` (macOS).
- Lists/loads/saves/deletes named sessions. Each session is a snapshot of the same state-blob format.
- User can load a session into the current plugin instance — replaces the in-memory state blob.

## JS ↔ C++ Bridge Protocol

### Request/response (JS calls C++ via `Juce.getNativeFunction(name)`)
Registered in `PluginEditor` constructor via `.withNativeFunction("name", lambda)`. Lambda runs on the message thread; uses `juce::MessageManager::callAsync` if it needs to dispatch to the audio thread.

Native functions (full list in [parameter-spec.md section 5](parameter-spec.md)):
- Audio: `playChord`, `playProgression`, `stopAll`, `setInstrument`
- State: `requestInitialState`, `pushState`
- Sessions: `listSessions`, `loadSession`, `saveCurrentSession`, `deleteSession`, `renameSession`, `duplicateSession`
- MIDI: `exportMidi`
- Diagnostics: `getBuildInfo`

### Push (C++ pushes via `WebBrowserComponent::emitEventIfBrowserIsVisible`)
Editor `juce::Timer` at 30 Hz reads atomic state from processor, emits:
- `currentBeatChanged`, `playStateChanged`, `hostBpmChanged`, `previewNotes`, `error`

JS side subscribes via `Juce.addEventListener("name", callback)`.

## Thread-Safety Strategy

| Data | Owner | Reader | Mechanism |
|---|---|---|---|
| APVTS parameter values | APVTS | audio thread (read via `getRawParameterValue()->load()`) | lock-free atomics |
| Scheduler current beat | audio thread | message thread (30 Hz timer in editor) | `std::atomic<int>` |
| Current state blob (XML) | message thread | message thread only | `juce::CriticalSection` (rare contention — only on save/load/state push) |
| Sampler sound set swap | message thread | audio thread | swap via `juce::Synthesiser::clearSounds()` + `addSound()`; gated by `MessageManagerLock` only on instrument change |
| Session file I/O | message thread | n/a | regular blocking file ops, off audio thread |
| MIDI byte stream from JS | message thread | message thread (FileChooser) | passed as `juce::var` (binary data) |

**Audio thread invariants:**
- No allocations (`juce::ScopedNoDenormals`, pre-allocated voice buffers)
- No locks (atomic params, lock-free swap for instrument)
- No file I/O (sessions/MIDI are all message-thread)
- Sampler sound swap deferred: `setInstrument` message-thread call sets a "pending instrument id" atomic; audio thread polls at top of `processBlock` and swaps if changed using `juce::SpinLock` (sub-microsecond)

## Complexity Assessment

**Score: 5 / 5 (Research / Expert tier — but de-risked)**

### Why 5
- Multi-thread coordination: audio + message + WebView, with state synced across all three
- Sample-based instrument engine with mid-flight instrument switching
- Host-transport sync with mode switching
- File-based session library + DAW state blob coexisting
- 14 React components, ~3,000 lines of mood-engine logic, 4 sampled instruments — full feature parity with a mature Electron app
- File I/O subsystem (cross-platform paths, JSON round-trip)
- Bundle pipeline: separate Vite build → BinaryData embed → CMake glue

### Why not 6 (de-risking via Phase 0 spike)
- Vite+React 19+JUCE WebView2 boot is already proven end-to-end in FL Studio
- JS↔C++ native function bridge round-trip is already proven
- @dnd-kit reorder is already proven
- Resource provider footgun is already neutralized (single-file bundle)
- Sampler quality is the biggest remaining unknown — handled in Sub-phase D below

### Implication for implementation
**Phased implementation, 7 sub-phases (A–G). Single-pass impossible at this scope.**
